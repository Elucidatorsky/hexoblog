作用域

1. 作用域是什么？

我们可以简单的把作用域理解成代码执行的一块区域。

作用域的责任是在这个区域内收集并维护由变量组成的一系列查询，根据一套严格的规则，限制执行代码对这些变量的访问权限。

2. 作用域如何发挥作用？

要理解作用域如何发挥作用，我们先要了解一下代码的编译处理。

参与处理的角色如下：

引擎 从头到尾负责JS程序的编译及执行（执行者）

编译器 负责语法分析及代码生成（阐明执行措施）

作用域 负责收集维护变量并确认当前执行代码对变量的访问权限（告知执行的地点和规定）

当你写了一行代码，例如var a = 2;

在编译器中，这句话会被分解成var a和a = 2

遇到var a，编译器会先向当前所在的作用域询问十分已经有一个该变量名称存在。如果有，编译器会忽略该声明，继续进行编译。如果没有，在当前作用域声明一个新的变量，命名为a。

接下来编译器会为引擎生成运行所需的代码，这些代码用来处理a=2这个赋值的操作。引擎会先询问作用域，在当前作用域集合中十分有一个叫做a的变量。如果有，就使用该变量，没有，向上一级作用域继续查找该变量。

如果引擎最终找到了a变量，就会将2赋值给它。否则会抛出一个ReferenceError异常。

3. 作用域的嵌套

当一个块或函数嵌套在另一个块或者函数中时，就发生了作用域的嵌套。因此，在当前作用域无法查找到某个变量时，引擎就会在外层的作用域继续查找，直到找到该变量，或者到达最外层的作用域（全局作用域）为止。而在外部的作用域无法向内部作用域查找变量。

4. LHS和RHS

但在javaScript里面存在一种比较特殊的情况。

在变量还未声明（任何作用域都无法找到该变量）的情况下，直接对变量进行赋值操作，全局作用域就会创建一个具有该名称的变量，将其返还给引擎（非严格模式下）。

这里就涉及到了LHS和RHS的赋值操作了，让我们简单的了解一下。

LHS 赋值操作的左侧 Left Hand Side；查找的目的是对变量进行赋值。
RHS 赋值操作的右侧 Right Hand Side；查找的目的是获取变量的值。

LHS和RHS查询都会在当前的执行作用域中开始，如果未找到所需的标识符，就会向上级作用域继续查找目标标识符，最后抵达全局作用域。

不成功的RHS引用会导致抛出ReferenceError异常。不成功的LHS引用会导致自动隐式地创建一个全局变量（非严格模式下），该变量使用LHS引用的目标作为标识符。

再让我们看看之前的代码，当进行b=2操作时，对b进行了LHS查询，因为实际上我们并不关心当前b的值是什么，只是想为=2寻找一个赋值目标。但并没有b这个变量的声明，所以会自动隐式的创建一个全局变量。

进行console.log(b);和console.log(a);操作时，我们需要知道a和b变量的值，进行RHS查询，变量a并没有声明，抛出了ReferenceError异常，但b因为隐式声明过了，所以能正常查询到b变量，输出2。

5. 词法作用域

5.1 词法化

大部分语言编译器的第一个工作阶段叫做词法化。词法化会对源代码中的字符进行检查，如果是有状态的解析过程，还会赋予单词语义。

简单的来说，词法作用域就是定义在词法阶段的作用域。也就是写代码期间函数所声明的位置来定义的作用域。

考虑以下代码：

在这个例子中有三个逐级嵌套的作用域。

function foo( a ) { ①
var b = a * 2; ②
function bar( c ){
console.log(a,b,c); ③
}
bar(b*3);
}
foo(2); // 2,4,12

① 包含了整个全局作用域，只有一个标识符：foo

② 包含着foo创建的作用域，其中有三个标识符： a、bar、b

③ 包含着bar创建的作用域，只有一个标识符：c

作用域层级由其所对应的作用域块代码写在哪里决定，他们是逐级包含的。

查找

作用域从内向外查找，在上述代码中，它首先从bar()所在作用域寻找，无法在当前作用域找到a、b变量，继续向外查找，在foo()作用域中找到a、b，b由a*2得到赋值，获得所需变量信息后，不会再向外部作用域继续查找。

作用域查找会在找到的第一个匹配的标识符时停止。在多层嵌套作用域中可以顶i一同名的标识符，但内部的标识符会“遮蔽”外部的标识符。

全局变量会自动成为全局对象的属性，因此可以间接的通过对全局对象的引用来对其访问。

例如浏览器中的windows对象。windows.a可以访问那些被同名变量所遮蔽的全局变量

无论函数在哪里被调用，无论如何被调用，词法作用域都只由函数被声明所处的位置决定。

词法作用域只会查找一级标识符，比如代码中使用了foo.bar.baz，词法作用域查找只会查找foo标识符，找到变量后，在对其内部进行引用。

5.2 欺骗词法

如果我们想在运行时改变词法作用域，我们改如何去做呢？

JavaScript中有两个机制来实现这个目的。社区普遍认为加入这个机制不是声明好主意。其中很重要的一个点是：欺骗词法作用域会导致性能下降

5.2.1 eval

eval()函数可以接受一个字符串作为参数，可将参数内容视为存在于当前eval()所处位置的代码，就好像代码是写在那个位置一样。

当foo()内部的console.log(b);被执行时，会在foo()内部直接找到b，覆盖了外部作用域的b，只会输出“3“。

在这个例子中，为了简洁表示eval()的作用，传递的参数是固定的字符串，在实际应用中，eval()通常被用来执行动态创建的代码

5.2.2 with

十分不推荐使用with关键字来欺骗词法作用域，它会很容易导致不少问题的发生。

with通常被当作重复引用同一个对象中的多个属性的快捷方式，可以不需要重复引用对象本身。

但with操作有时候会导致一些问题。

使用with赋值对象内部的属性时，当对象内部没有with操作赋值的属性时，会造成不成功的LHS查找自动创建新的变量。
with可以将对一个或多个属性的对象 处理为一个完全隔离的(新的)词法作用域。这个对象的属性也会被处理为定义在(写在) 这个作用域中的词法标识符。

with块内部的var声明不会被限制在with内部的词法作用域中，而是被添加到with所处的函数作用域中。

5.2.3 性能

虽然eval()和with能使代码有更好的扩展性，能实现更复杂的功能。但整体性能上会受到很大影响。

JavaScript引擎会在编译阶段进行数项的性能优化，预先确定所有变量和函数的定义位置。如果引擎在代码中发现了eval()或with，它只能简单地假设关于标识符位置的判断都是无效的。所有的优化可能都会失效，代码会运行的更慢。

6. 函数作用域

简单来讲，函数作用域就是属于函数内部的作用域。

函数作用域的含义是指，属于这个函数的全部变量都可以在整个函数范围内使用及复用（实际上在嵌套的作用域中也可以使用）。

6.1 隐藏内部实现

我们知道，作用域只能由内向外部访问，无法由外向内访问。我们可以通过使用函数将不给外部公开的变量以及内部函数方法包裹起来，隐藏内部的实现，将内容私有化，来防范外部操作对内部标识符的更改。

我们无法访问内部的doSomethingElse()函数以及变量b，有效的保证内部变量和函数不污染全局作用域，变量函数私有化也更加安全。

规避冲突

”隐藏“作用域中的变量和函数还可以避免同名标识符之间的冲突

i=3意外的覆盖了声明在for循环内的i，每次循环到bar方法中，使用LHS查询会查找到for循环内部的i变量，使得for循环作用域内的i永远是3。

我们只需要在作用域内先声明一个变量var i = 3”隐藏“内部变量，就可以避免冲突。

6.2.2 具名和匿名

没有名称标识符的函数方法叫做匿名函数。

在Javascript中函数表达式是可以匿名的【(fuction(){})属于函数表达式】，而函数声明不可以省略函数名。

虽然匿名函数写起来简单快捷，但也有几个缺点需要考虑。

  1. 匿名函数在栈追踪中不会显示处有意义的函数名，使得函数调试变得困难。
  2. 如果没有函数名，函数只能使用已经过期的arguments.callee引用自身。
  3. 匿名函数降低了代码可读性。

6.2.1 自执行函数

我们已经知道任意代码片段外部添加包装函数可以将内部信息”隐藏“起来，但这种解决问题的办法并不完美，不仅需要声明一个函数污染其所在的作用域，还必须将声明的函数进行调用才能运行其中的代码。

如果函数不需要函数名（或者至少函数名不污染所在作用域），并且能自动执行，这将会更理想。JavaScript提供了解决问题的方案。

7. 块作用域

{...}内部就是块作用域

我们在for循环和if语句中定义变量bar，通常是因为想在其内部使用bar，但是当我们使用var声明一个变量时，使用的变量必会影响到外部作用域。

Javascript在ES6引入的关键字let将变量限制在块作用域中，解决了这个问题。

7.1 let

let关键字可以将变量绑定到所在的任意作用域中（通常是{...}内部）。外部无法访问其内容。

注意：let声明不会在块作用域中进行变量提升。声明的代码被运行之前，引擎无法找到被声明的变量。

垃圾收集

另一个块作用域非常有用的原因和闭包以及回收内存垃圾的回收机制有关。

一般情况下，当函数执行形成一个私有的作用域，当执行完成后就销毁了。我们可以用块作用域把执行后用不到的代码包裹起来{..}，执行完进行垃圾回收节省内存空间。
